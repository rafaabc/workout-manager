# Workout Manager

Full-stack application for workout management, user authentication, calendar scheduling and performance metrics.

Built by a **QA Engineer** exploring how **generative AI** can accelerate the development and testing of applications. The project puts into practice a **multi-layer test automation strategy**, from isolated unit tests to performance testing.

## Objective

Provide a workout tracking system (REST API + browser frontend) while exercising **test automation at multiple layers**, following the test pyramid approach:

| Layer | Tool | Location |
|---|---|---|
| Backend Unit Tests | Node.js Test Runner + `assert` | `test/unit/` |
| Integration Tests | Node.js Test Runner + `assert` | `test/integration/` |
| Frontend Unit Tests | Jest + jsdom | `frontend/test/unit/` |
| API Tests | Mocha + Chai + Supertest | `test/api/` |
| Performance Tests | k6 | `performance/k6/` |
| Code Coverage | c8 | — |
| Static Analysis | SonarCloud + ESLint | CI pipeline |

## Test Strategy

The test architecture follows the **test pyramid** principle — a broad base of fast, isolated unit tests supporting fewer (but critical) API and performance tests:

```
        ┌───────────────┐
        │  Performance   │  k6 — load & stress testing
        ├───────────────┤
        │  API Tests     │  Mocha + Chai + Supertest — HTTP contracts,
        │                │  status codes, auth flows, business rules
        ├───────────────┤
        │  Integration   │  Node.js Test Runner — repository layer,
        │                │  SQL queries, constraints, referential integrity
        ├───────────────┤
        │  Unit (front)  │  Jest + jsdom — UI logic, validations,
        │                │  API service mocks, DOM rendering
        ├───────────────┤
        │  Unit (back)   │  Node.js Test Runner + assert — services,
        │                │  models, validators in isolation
        └───────────────┘
```

| Layer | What it validates | Runs in CI |
|---|---|---|
| **Unit (backend)** | Business rules in services, models and validators — no HTTP or I/O dependencies | ✅ |
| **Unit (frontend)** | UI logic, input validation and API service integration using a simulated DOM (jsdom), without real API calls | ✅ |
| **Integration** | Repository layer against an in-memory SQLite database — SQL queries, UNIQUE/FK constraints, data integrity | ✅ |
| **API Tests** | HTTP contracts, status codes, request/response payloads, JWT authentication and end-to-end flows against a running server | ✅ |
| **Performance** | Behavior under load with k6 — identifies bottlenecks and capacity limits | Local only |

Coverage reports are generated by **c8** (Istanbul/V8) for every test layer, ensuring visibility into tested and untested code paths.

## Database

The project uses **SQLite** as its persistence layer via [better-sqlite3](https://github.com/WiseLibs/better-sqlite3).

### Why SQLite?

- **File-based** — no external database server required
- **Zero configuration** — works out of the box
- **Lightweight** — ideal for small-to-medium applications
- **In-memory mode** — enables fast, isolated test execution

### Initialization

On application startup, `src/database/database.js` opens a connection, enables foreign keys, and runs any pending SQL migrations from `src/database/migrations/`.

### Storage

| Environment | Database Path |
|---|---|
| Production | `database/workout-manager.db` (file-based, persistent) |
| Tests | `:memory:` (in-memory, ephemeral) |

The `database/` directory is git-ignored so local data is never committed.

### Repository Layer

A **repository layer** (`src/repositories/`) encapsulates all SQL queries. Services never execute SQL directly — they delegate data access to repositories, which receive the database instance via dependency injection.

Architectural flow:

```
Controller → Service → Repository → Database (SQLite)
```

| Repository | Responsibility |
|---|---|
| `userRepository.js` | User creation and lookup |
| `workoutRepository.js` | Workout CRUD and aggregation queries |
| `goalRepository.js` | Annual goal upsert and retrieval |

## Getting Started

### Prerequisites

- **Node.js** ≥ 21
- **npm** (comes with Node.js)
- **k6** (only for performance tests) — `npm install -g k6`

### Installation

```bash
npm install
```

### Starting the Backend

```bash
npm start
```

The API will be available at `http://localhost:3000` and the Swagger documentation at `http://localhost:3000/api-docs`.

### Starting the Frontend

```bash
npm run server
```

Serves the `frontend/` directory via `http-server`. Requires `http-server` installed globally (`npm install -g http-server`) or use `npx http-server frontend`.

## Running Tests & Reports

### Backend Unit Tests

```bash
# Run tests
npm run test:backend:unit

# Run tests with code coverage
npm run test:backend:unit:coverage
```

Coverage is generated by **c8** and printed to the terminal. An HTML report is written to the `coverage/` directory.

### Integration Tests (Repositories)

```bash
# Run tests
npm run test:integration

# Run tests with code coverage
npm run test:integration:coverage
```

Runs repository tests against an **in-memory SQLite** database — no file or server required.

### Frontend Unit Tests

```bash
# Run tests
npm run test:frontend:unit

# Run tests with code coverage
npm run test:frontend:unit:coverage
```

Coverage is generated by **c8** and printed to the terminal.

### API Tests

```bash
# Run tests (spec reporter)
npm run test:api

# Run tests with Mochawesome HTML report
npm run test:api:report
```

Tests run against an **in-memory SQLite** database — no running server required.
The HTML report is generated under `mochawesome-report/index.html`.

### Performance Tests (k6)

> **Requires** k6 installed globally and the API server running.

```bash
# Run performance test (terminal output)
npm run k6:register

# Run performance test with HTML report
npm run k6:register:html
```

The HTML report is saved to `performance/k6/login/register_user_performance.html`.

### Code Quality

```bash
npm run lint          # Static analysis (ESLint + security rules)
npm run lint:fix      # Auto-fix ESLint issues
npm run format        # Auto-format with Prettier
npm run audit         # Dependency vulnerability scan
```

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| `POST` | `/api/users/register` | — | User registration |
| `POST` | `/api/users/login` | — | Login (returns JWT) |

## Business Rules

### 1) User Registration and Login

- Registration requires a unique username and a password with at least 8 characters (letters + numbers).
- Login validates credentials and returns a JWT token.
- The user remains authenticated until logout or session expiration.

> **Test coverage:** `test/api/login/` · `test/unit/services/userService.test.js` · `test/unit/utils/validators.test.js`

### 2) Scheduling Workouts on the Calendar

- Only authenticated users can access the calendar.
- The calendar displays the current month with navigation to previous/next months.
- Clicking a day marks or unmarks a workout; marked days are visually highlighted.
- One workout per day per user; records persist across sessions.

> **Test coverage:** `test/api/calendar/` · `test/unit/services/workoutService.test.js` · `frontend/test/unit/components/calendar.test.js`

### 3) Planned vs. Completed Workouts Metrics

- Users can set an annual workout goal.
- The system calculates workouts completed per month and year, plus the percentage toward the annual goal.
- Metrics update automatically when workouts are marked/unmarked and are scoped per user.

> **Test coverage:** `test/api/metrics/` · `test/unit/services/metricsService.test.js` · `frontend/test/unit/components/metrics.test.js`
